## TL;DR

`learn from TICKET-X` should resolve its investigation by traditional exact search. SIO-1134 solved this for tickets created via the chat Create-ticket button (the endpoint links `Incident.ticketKey` at creation). The remaining gap: reports pasted BY HAND into hand-created Jira tickets carry no machine-readable key. Fix: stamp `**Request-Id:** <state.requestId>` into every report's header block, and scan fetched ticket text for it in the learn lane -- when the id exists in the KG, link directly with zero embeddings. Success = a hand-pasted report round-trips to an exact match.

**Read SIO-1134 first**: partially supersedes this ticket. Do NOT duplicate the ticket-creation path; this is only for the manual-paste lane.

## Context -- how this ticket came to be

Third phrasing of the same user ask during the HIL learning sessions: "i still dont know why we cannot do a traditional search for the actual ticket". The blocker was data, not search: for most tickets the internal key exists in neither system. SIO-1130 (auto-confirm), SIO-1132 (exec-summary embedding) and SIO-1134 (curated ticketKey) landed; this is the last lane. Design: `docs/superpowers/specs/2026-07-16-hil-learning-loop-design.md`; the full intent is in the Linear description (kept current).

## Where the bodies are buried

1. **KG incident id IS the requestId** -- `packages/agent/src/graph-knowledge.ts:100` (`recordGraphEntities` merges `Incident {id: state.requestId}`). So a Request-Id found in ticket text is directly a KG node id. Lookup by node id: reuse the pattern of `incidentByTicketKey` in `packages/knowledge-graph/src/reader.ts` (`MATCH (i:Incident {id: $id}) RETURN ...` single row) -- check whether a by-id reader already exists before writing one.

2. **Report header is LLM-assembled** -- `packages/agent/src/aggregator.ts:227` instructs the model to render the `Generated` header field; `rawAnswer` comes back at `aggregator.ts:894-896` and the final message is emitted at `aggregator.ts:1112`. Do NOT rely on a prompt instruction for the Request-Id -- LLM-stamped values drift (see memory `reference_llm_year_drift_needs_deterministic_guard`). Append deterministically to `finalAnswer` before the AIMessage is built, e.g. a footer line `**Request-Id:** ${state.requestId}`.

3. **Learn-lane match order** -- `packages/agent/src/learn/match.ts:90` (curated `incidentByTicketKey` first), `:134` (ticket-mention pin), `:147` (vector). The Request-Id scan slots BETWEEN the curated lookup and the pin.

4. **Gate auto-confirm order** -- `packages/agent/src/learn/ticket.ts:265-291` (`learnMatchGate`): `ticket-link` candidate short-circuits first, then single `ticket-mention` pin, then zero-candidates create, else interrupt.

5. **`via` enum is shared + rendered** -- `packages/shared/src/hil-learning.ts` (`HilMatchCandidateSchema.via: z.enum(["vector","ticket-mention","ticket-link"])`), passthrough whitelist in `apps/web/src/lib/server/sse-pump.ts` (`emitHilLearningInterrupt` maps unknown via to "vector"), label in `apps/web/src/lib/components/LearningMatchCard.svelte:57-61`.

## The fix (step-by-step)

1. `packages/agent/src/aggregator.ts`: append the Request-Id footer line deterministically where `finalAnswer` is assembled (~`:1112`). Keep it in the header/footer block style the reports already use (`**Generated:**`, `**Ticket:**`).
2. `packages/agent/src/learn/match.ts`: after the curated-ticketKey miss, regex the fetched ticket text (description + comments, ALREADY flattened by `learn/ticket.ts`) for `Request-Id:\s*([0-9a-f-]{36})`; on hit, `MATCH (i:Incident {id})`; if present, return a single candidate `via: "request-id"` (empty `hilTicketEmbedding`, like the ticket-link path).
3. `packages/agent/src/learn/ticket.ts` (`learnMatchGate`): auto-confirm a `request-id` candidate exactly like `ticket-link` (`:265`). On apply, the existing SIO-1134 curation writes `ticketKey`, so the NEXT learn resolves via ticket-link without re-scanning.
4. `packages/shared/src/hil-learning.ts`: add `"request-id"` to the via enum. Update the sse-pump passthrough and the LearningMatchCard label ("report Request-Id found in ticket").
5. Tests: match unit tests (hit, miss, malformed uuid, id not in KG -> falls through to pin/vector), gate auto-confirm test, aggregator footer test (deterministic value equals `state.requestId`), fixture with a sanitized pasted report.
