## The migration pattern (canonical)

For a file with the standard shape (~80% of remaining files):

```typescript
// 1. Promote underscored type alias to non-underscore so we can use it.
- type _FooParams = z.infer<typeof fooValidator>;
+ type FooParams = z.infer<typeof fooValidator>;

// 2. Tighten McpError factory's details parameter.
  function createFooMcpError(error, context: {
    type: ...;
-   details?: any;
+   details?: unknown;
  }): McpError { ... }

// 3. Type the handler arg.
- const fooHandler = async (args: any): Promise<SearchResult> => {
+ const fooHandler = async (args: FooParams): Promise<SearchResult> => {

// 4. Collapse the duplicated inputSchema (optional but recommended).
- inputSchema: {
-   field1: z.string()...,
-   field2: z.boolean()...,
- },
+ inputSchema: fooValidator.shape,
```

## Variations to watch for

**Duration fields with `as any`** (seen in `tasks/*.ts`):
```typescript
- timeout: z.union([z.string(), z.number(), z.literal(-1), z.literal(0)]).optional(),
+ // Elasticsearch Duration accepts string ("30s") or special values -1/0; not arbitrary numbers.
+ timeout: z.union([z.string(), z.literal(-1), z.literal(0)]).optional(),
```
Then drop the `params.timeout as any` -> `params.timeout` in the SDK call. The `as any` was hiding the schema-vs-Duration mismatch.

**Schema name varies**:
- `<noun>Validator` (most common)
- `<Noun>Params` (some files, e.g., `ListTasksParams`)
- `<noun>Schema` (a few files)
Always grep first: `grep -E "const \w+(Validator|Params|Schema)\s*=\s*z\.object" <file>`.

**Files with non-standard handler signatures**:
- `(server, esClient: Client)` -- missing type on first param. Add `: McpServer`.
- `async (toolArgs: any, _extra: any)` -- both args are `any`. Type both: `args: FooParams, extra: Record<string, unknown>` (or just leave `extra` and only type `args`).

**Files that don't follow the registerTool pattern** (e.g., `tasks/index.ts`, `cancel_task.ts` exports object literals): handler signatures already used `z.infer<typeof X>` -- only collapse the inputSchema and remove `as any` casts.

**Files with no validator at all**: a handful of registry/index files have `args: any` but no Zod schema. Type as `unknown` instead of `z.infer<typeof X>`. Verify with grep before assuming a validator exists.

## Workflow per file

1. `Read` the file fully (or just the parts with `args: any`, `details?: any`, the validator declaration, and `inputSchema:`).
2. Apply the 4 edits above.
3. After every 4-5 files: `bun run --filter '@devops-agent/mcp-server-elastic' typecheck` to catch surfaced mismatches early.
4. If typecheck surfaces a real type mismatch (like the `waitForActiveShards` fix in PR2 or the `Duration` fix in PR3.3), fix the schema -- do not add a new `as any`.

## Why this was bigger than the original plan

The original plan estimated PR3 as a single ~80-file PR. Reality:
- Type-alias tightening cascade was tiny (1 file errored on typecheck) -- the loose `(server: any, esClient: any)` was masking *handler arg* problems, not registration problems.
- The bulk of the work is mechanical handler retyping that biome already flags as warnings, not typecheck errors.
- Each file has 3-4 small edits, but with the canonical pattern requiring file inspection (validator name varies, inputSchema may diverge from validator, some files have `as any` casts on Duration fields that hide real mismatches).
- I attempted a regex-based codemod and it produced bad output (duplicate type aliases, mid-function-body insertions). Reverted; recommended a ts-morph codemod for the next session if hand-editing remains slow.
