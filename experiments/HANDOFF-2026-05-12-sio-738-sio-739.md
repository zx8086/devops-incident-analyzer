## SIO-739: Pipeline hangs after `validate` (HIGH)

### Repro evidence

Run at 2026-05-12 12:00 UTC, requestId `c4c0d8c9-c604-429e-accb-ce51af20ea34` (LangSmith trace available under project `devops-incident-analyzer`).

```
12:00:38 info: Sub-agent completed (elastic correlation, 223s duration)
12:00:38 info: Checking confidence against HITL threshold {confidenceScore:0.59,threshold:0.6}
12:00:38 warn: Confidence below threshold, flagging for user review
12:00:38 info: Validating answer {answerLength:12158,retryCount:0}
12:00:38 info: Validation passed
(no further log lines for 25+ minutes; frontend never received a response)
```

Expected next log lines per the 13-node pipeline definition in CLAUDE.md:

```
Mitigation steps generated {...}
Generated follow-up suggestions {...}
```

Neither appeared. The answer (12,158 chars) was already aggregated and validated — it just never reached the client.

### Why this is interesting

Earlier in the same session, an 11:55 UTC run with `confidence: 0.72 (capped to 0.59)` completed the full pipeline (mitigation + follow-up emitted). The hung run had `confidence: 0.59` natively (below 0.6 threshold). The confidence-gate emitted `Confidence below threshold, flagging for user review` — which suggests the HITL flag may interact with downstream nodes in a way that doesn't advance the graph.

**Hypothesis:** the `checkConfidence` node's HITL flag may route to a node that hangs (e.g. `proposeMitigation` calling Bedrock without a wall-clock timeout, or an edge that never resolves when `requiresHumanReview` is set).

### Investigation paths

1. **Read `packages/agent/src/graph.ts`** — look at the `checkConfidence` node's conditional edges. When `confidenceScore < threshold`, does it route to `proposeMitigation` like the happy path, or to a different node? Look for an HITL-specific branch.

2. **Check `proposeMitigation` for timeouts** — `packages/agent/src/nodes/propose-mitigation.ts` (path approx). Does the Bedrock LLM call have an `AbortSignal.timeout(...)`? If not, that's the smoking gun.

3. **Pull the LangSmith trace** — the in-flight node at the freeze point will be visible. Use `langsmith-fetch traces /tmp/traces --limit 5 --include-metadata` filtered to project `devops-incident-analyzer` and the requestId above.

4. **Check the SSE plumbing** — `apps/web/src/routes/api/chat/+server.ts` or wherever the agent stream is wired. If the frontend disconnected and the server is blocked on `res.write()` without backpressure handling, this is the cause.

### Acceptance criteria from SIO-739

- After `validate` passes, the pipeline always advances to `proposeMitigation` within 60s, or routes elsewhere with a clear log line.
- Any LLM-calling node has a wall-clock timeout that fails the run with a clear error rather than hanging.
- The frontend receives either a completed response or a clear failure within the pipeline's deadline.

## SIO-738: Kafka sub-agent filter excludes `restproxy_*` / `connect_*` REST tools (MEDIUM)

### Repro evidence

Same 12:00 UTC run, but the issue is visible earlier at 11:53:11:

```
Creating ReAct agent with tools {toolCount:25, totalTools:55, filtered:true}
```

Across 47 tool invocations in the kafka sub-agent, **zero** were `restproxy_*` and **zero** were `connect_*` REST tools. The runtime DOES have `RESTPROXY_ENABLED=true` and `CONNECT_ENABLED=true` per memory `reference_kafka_mcp_agentcore_ksql_disabled.md` — so 9 `restproxy_*` and 9 `connect_*` tools are registered server-side. The action-driven filter just didn't pick any for this prompt.

The generated report's Gaps section says:

> Kafka Connect REST API not probed: no `connect_*` tool available.

That's wrong — the tools exist; they were filtered out client-side.

### Investigation paths

1. **Read `agents/incident-analyzer/agents/kafka-agent/tools/`** — the YAML tool catalogs and action mappings. Find which actions surface `restproxy_*` and `connect_*` tools. Are there any?

2. **Read the action classifier** — `packages/gitagent-bridge/` likely has the prompt-to-action mapping. When a user prompt names "REST Proxy", does any action fire? Same question for "Connect" / "Kafka Connect".

3. **Test fix candidate** — add an action like `rest_proxy_health` whose tool list includes `restproxy_get_brokers`, `restproxy_get_topics`, etc. Re-run the same prompt and verify these tools appear in the trace.

### Acceptance criteria from SIO-738

- A query naming "REST Proxy" surfaces a `restproxy_*` probe in the agent's tool calls.
- A query naming "Connect" surfaces `connect_get_cluster_info` or `connect_list_connectors`.
- The Gaps section no longer claims these REST APIs are unavailable when they ARE registered server-side.
- Non-REST-Proxy / non-Connect prompts still see the same 25-tool budget as today.

### 2. 30s per-call deadline may be tight for cold-starts

During the 11:53:52 / 11:53:56 cluster in the smoke run, three concurrent `ksql_*` calls hit TCP timeouts (30s) then bailed via the JSON-RPC deadline guard with `gaveUpDueToDeadline: true, totalMs: 33194`. That's the SIO-737 budget working as designed — but if you find cold-starts routinely exceed 30s before the runtime is ready, the per-call deadline could be relaxed.

Today it's a module-scope constant in `packages/shared/src/agentcore-proxy.ts`:

```ts
const JSONRPC_RETRY_DEADLINE_MS = 30_000;
```

SIO-737's spec marked env-var configuration of this as **out of scope** for that PR. If this becomes a real pain, a small follow-up could lift it to env (`AGENTCORE_PROXY_DEADLINE_MS=60000`) without changing the default. Don't do it preemptively — wait for evidence it bites.
