## Gotchas worth remembering

1. **The OAuth provider tests previously nuked real user tokens.** The Atlassian `test/oauth-provider.test.ts` on `main` (pre-PR) had `afterEach(() => rmSync(~/.mcp-auth/atlassian, { recursive: true }))`, which deleted the developer's actual persisted tokens on every test run -- forcing a re-seed each time the test suite ran. The new GitLab test inherited the same footgun. **Both fixed in commit `fd73fb0`** (synthetic test endpoints `gitlab.test.invalid` / `mcp.atlassian.test.invalid` so cleanup only removes the test-specific file). If you add a new MCP-server OAuth test, follow the synthetic-endpoint pattern.

2. **The Atlassian OAuth dropdown can't be auto-selected from our side.** Atlassian's MCP DCR contract does not expose a documented `site` or `audience` query parameter to pre-pin the workspace in the authorize URL. The dropdown is server-rendered. We rely on `ATLASSIAN_SITE_NAME=pvhcorp` to pin the post-OAuth `getAccessibleAtlassianResources` resolution -- so even if the user mis-clicks, our code throws a clear `'site "pvhcorp" not found'` error. The dropdown only fires the very first time you seed (or after deleting `~/.mcp-auth/atlassian/*.json`); refresh tokens reuse the choice.

3. **Stale-registration auto-discard is silent and one-shot.** When the `BaseOAuthClientProvider` loads a persisted `clientInformation` whose `token_endpoint_auth_method` differs from the current code's expectation (e.g. legacy `client_secret_post` row from before SIO-685 vs current `none`), it logs a warning, unsets the row, persists, and returns `undefined` so the SDK re-registers. **Skipped when `persistedMethod` is `undefined`** (legacy fixtures that pre-date storing the method) -- otherwise the existing Atlassian test would break.

4. **`writeFileSync` mode option only fires on file creation.** Node-level gotcha. The base provider does `writeFileSync(path, data, { mode: 0o600 })` AND `chmodSync(path, 0o600)` to enforce 0o600 on pre-existing files written by an older version. Don't remove the chmod -- it's the migration path.

5. **`isHeadless()` checks both env AND `process.stdout.isTTY === false`.** This catches the case where someone forgot to set `MCP_OAUTH_HEADLESS=true` on a headless server. The seed CLIs explicitly `delete process.env.MCP_OAUTH_HEADLESS` so they always run interactively.

6. **The MCP SDK's auth flow tries `refresh_token` first, then falls back to fresh authorize.** Confirmed by reading `node_modules/.bun/@modelcontextprotocol+sdk@1.29.0/.../client/auth.js:230-312`. As long as `tokens.refresh_token` is on disk, the SDK silently refreshes; you only see the popup again if the refresh itself fails. `expires_in: 7200` means GitLab access tokens are valid 2 hours, refresh tokens currently never expire on GitLab (per their docs and `gitlab-org/gitlab#595536`).

7. **Atlassian provider keeps `client_secret_post`.** Atlassian's DCR endpoint actually returns a real `client_secret`, so confidential-client auth works there. Don't switch Atlassian to `none` "for parity" -- it would regress.

8. **The eval pipeline's precheck script is localhost-only** (per the prior handoff's gotcha 7). It probes `localhost:9080-9085`. AgentCore-deployed MCP servers will report missing even though the actual MCP URLs in `.env` may work -- bypass with a temp script if needed.

9. **The seed CLI auth URLs are public for one-time use.** The auth URL contains a one-shot authorization code request; once redeemed (or after a few minutes) it can't be reused. Don't worry about leaking the URL in logs/screenshots after the seed completes.
