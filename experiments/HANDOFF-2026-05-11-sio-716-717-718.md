## Story

The investigation started from a c72 Kafka health report (2026-05-11T15:41 UTC) that claimed ksqlDB was hard down with `statusCount: {RUNNING: 1, UNRESPONSIVE: 1}` across all 29 persistent queries. Looked credible. Wasn't.

Sequence of revelations, each overturning the previous:

1. **Round 1 (wrong):** "Agent hallucinated the data — `KSQL_ENABLED` is unset on AgentCore." Disproven by inspecting the live runtime config you pasted: KSQL_ENABLED=true, real endpoints set.
2. **Round 2 (partial):** "Data is real, but the LLM under-reported." LangSmith trace `019e17b2-2bf1-75e5-92cb-68a03df0c4c5` confirmed 7 successful tool calls returned real data. Still missing: live state.
3. **Round 3 (correct):** Probed the live AgentCore MCP through the SigV4 proxy at `localhost:3000`. ksqlDB / Connect / Schema Registry all returning HTTP 503 (nginx default empty-upstream page). MSK direct (TCP 9092 to brokers) keeps working — separate network path.
4. **The pivot:** You pointed me at the Elastic synthetic monitor `b40a84fc-82ba-412c-bf82-faf828a946fd` for `https://ksql.prd.shared-services.eu.pvh.cloud/healthcheck`. UP. The agent's failing endpoints are `*.dev.shared-services.eu.pvh.cloud`. So: services are healthy on `prd`, agent is calling `dev`, c72 is production. **Environment misrouting.**

That last step is what split the work into SIO-716 (env-var fix, ops) and SIO-717 (the agent should have caught this itself via synthetic cross-check, engineering).

## What's blocking SIO-717's end-to-end verification

The SIO-717 code is correct and unit-tested, but the synthetic-cross-check rule path only fires when at least one tool error contains a hostname matching `(ksql|connect|schemaregistry|restproxy)\.(dev|prd)\.shared-services\.eu\.pvh\.cloud`. The hostname comes from the nginx 503 body, which **does not include the hostname** — only the generic `<html>503 Service Temporarily Unavailable</html>` page. So Rule 3 (the synthetic cross-check) does not trigger on today's traffic, even though the prompt-side rules (1 and 2 in SOUL.md) are exercised. End-to-end validation needs either:

- SIO-716 to ship so the path stops returning 503s entirely (synthetic cross-check is then dormant by design — happy path), OR
- A follow-up to pass the configured upstream hostname from the kafka MCP server through to the tool-error payload so the rule has something to match on. Worth a small follow-on ticket if you want belt-and-braces verification.

## Gotchas worth remembering

- The proxy log shows the tool outcome on each call as a single field. When the AgentCore HTTP envelope is non-2xx the line includes a `(http <code>)` suffix; otherwise the absence of an HTTP field means the envelope was 200. (Earlier sessions used `outer=`/`inner=` — that vocabulary is gone.)
- Synthetic monitors are independent ground truth — check them before concluding a service is down from agent evidence alone. SIO-717 Rule 3 codifies this but it's also a manual debugging habit worth keeping.
- c72-shared-services-msk has no env suffix in its name. Don't assume `dev` from the absence — check the MSK ARN's account / tags, or just ask. We almost mis-scoped SIO-716 as a "Confluent stack outage" before you corrected to "production cluster mis-routed to dev endpoints."
- `bun --hot` reloads source on file change, but does NOT restart the process. The SigV4 proxy state (cached creds, session IDs) survives reloads.
