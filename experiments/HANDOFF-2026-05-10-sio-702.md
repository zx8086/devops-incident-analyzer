## The bug in one sentence

The MCP SDK's `StreamableHTTPClientTransport` runs its own internal refresh path (`refreshAuthorization()` in `node_modules/@modelcontextprotocol/sdk/dist/esm/client/auth.js:266`) that **bypasses** the SIO-698 `refreshInFlight` lock in `GitLabOAuthProvider.refreshTokens()` -- so under N parallel sub-agent tool calls, N parallel `POST /oauth/token` requests fire, GitLab rotates the refresh_token after the first success, the remaining N-1 use a now-revoked token, get `400 invalid_grant`, and the SDK's `auth()` catches it and calls `provider.invalidateCredentials('tokens')` (auth.js:152) -- wiping the freshly-rotated tokens from disk.

The fix moves the synchronization from `refreshTokens()` (which only your own code calls) to `tokens()` (which **everyone** calls, including the SDK transport).

- `refreshInFlight` (oauth-provider.ts:22): only protects `GitLabOAuthProvider.refreshTokens()`, called from `proxy.callTool` (proxy.ts:172). The SDK transport's own refresh path goes around it.
- The "single-shot retry" guard at `proxy.ts:158-174` fires AFTER the SDK's `auth()` has already wiped credentials.
- `MCP_OAUTH_HEADLESS=true` correctly refuses to pop a browser, but only after the wipe.

## Recommended fix (user-selected)

**Override `tokens()` to refresh-on-read when expired.**

The SDK awaits `tokens()` at auth.js:261, so making it async is wire-compatible. Pseudocode:

```ts
// SIO-702: refresh-on-read pushes the synchronization from refreshTokens()
// (which only our code calls) to tokens() (which the SDK transport also calls).
// All N parallel reads now share the same refreshInFlight promise; only one
// /oauth/token POST ever fires; GitLab cannot trigger the rotation race
// because there is only one refresher.
async tokens(): Promise<OAuthTokens | undefined> {
  const persisted = this.persisted.tokens;
  if (!persisted) return undefined;
  if (!this.isExpired(persisted)) return persisted;
  // Expired -- delegate to the locked refresh. refreshInFlight dedupes.
  return await this.refreshTokens();
}
```

`isExpired()` doesn't exist yet. Use `expires_in` from the persisted token + the time it was saved, with a small skew (e.g. 30s) to refresh proactively. Persist the save timestamp alongside the token (extend the persisted shape; today only `expires_in` is stored, which is meaningless without a base time). Be careful: `expires_in` is the lifetime in seconds at the moment of issue, not an absolute expiry.

Open question for the implementer to decide:
1. Add `obtainedAt: number` (epoch ms) to the persisted shape and use it for expiry math, OR
2. Try-and-refresh-on-401 instead of pre-checking expiry. Simpler but doesn't eliminate the wasted first 401 round-trip.

Recommendation: **(1) is cleaner** -- proactive refresh, no wasted request, the existing `refreshInFlight` lock dedupes the rare race window between the expiry check and the actual refresh.

## Test strategy

### Unit test (in `packages/mcp-server-gitlab/tests/gitlab-client/`)

`oauth-provider-rotation-race.test.ts`:
1. Construct a `GitLabOAuthProvider` with a mocked `fetch` that returns valid tokens after a controlled delay.
2. Pre-persist an expired token (set `obtainedAt` to long ago).
3. Fire 10 concurrent `await provider.tokens()` calls via `Promise.all`.
4. Assert: exactly 1 fetch was called; all 10 returned the same fresh access_token; only 1 `OAuth tokens saved` log line; zero `OAuth credentials invalidated` log lines.
5. Negative: pre-persist a refresh_token that the mocked fetch rejects with `400 invalid_grant`. Assert exactly 1 `OAuthRefreshChainExpiredError` thrown across all 10 callers (the in-flight promise rejection propagates).

### Integration test (live or skipped-by-default)

Run `bun run dev` for the gitlab MCP server, fire 10 concurrent `gitlab_search` calls via curl/script, watch the log for `OAuth credentials invalidated` (must be zero) and count `POST /oauth/token` lines (must be 1 if any refresh was needed). Document the procedure; skipping in CI is fine.
