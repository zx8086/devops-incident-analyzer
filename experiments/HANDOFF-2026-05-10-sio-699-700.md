## SIO-699 — `kafka_consume_messages` breaks AgentCore runtime health-check

### What we know

The tool consistently returns `-32010 Runtime health check failed or timed out` and **also takes the entire AgentCore runtime offline for ~30s afterwards**, blocking subsequent tool calls. One bad call breaks everything until the runtime recovers.

### Reproduction (live, deterministic)

```bash
curl -sN -X POST http://localhost:3000/mcp \
  -H 'Accept: application/json, text/event-stream' \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"kafka_consume_messages","arguments":{"topic":"example-topic","maxMessages":1,"timeoutMs":5000}}}'
```

Returns:

```json
{
  "jsonrpc": "2.0",
  "error": {
    "code": -32010,
    "message": "Runtime health check failed or timed out. Please make sure that health check is implemented according to the requirements here - https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-service-contract.html"
  },
  "id": "..."
}
```

Probed in the same session: 5 of 5 admin-metadata tools succeeded (`list_topics`, `describe_cluster`, `describe_topic`, `get_topic_offsets`, `list_consumer_groups`); `describe_consumer_group` and `get_consumer_group_lag` also worked. So it is **not** a generic AgentCore problem and **not** an IAM problem (those would manifest differently). The pattern is specific to tools that spawn a real Kafka consumer.

### Hypothesis to verify first

`kafka_consume_messages` (defined in `packages/mcp-server-kafka/src/services/kafka-service.ts`, search for `consumeMessages`) constructs a `@platformatic/kafka` consumer per call:
1. Create consumer with ephemeral group
2. Subscribe to topic
3. Poll until `maxMessages` reached or `timeoutMs` elapses
4. Tear down consumer

Steps 1–3 likely take long enough that AgentCore's runtime contract considers the runtime "unhealthy." The contract is at https://docs.aws.amazon.com/bedrock-agentcore/latest/devguide/runtime-service-contract.html — read this first; it dictates the exact ping path and cadence the runtime is expected to respond to.

### Where to start

Read in this order:
1. **AgentCore runtime contract** (link above) — figure out the actual health-check timeout window
2. `packages/mcp-server-kafka/src/services/kafka-service.ts` — find `consumeMessages` (or equivalent — the tool maps to a service method). Check whether the consumer runs *inside* the request-response loop (blocking) or somehow detached
3. Compare with `kafka_describe_consumer_group` in the same file — that tool DID work; what makes its admin-client call coexist with health-check pinging while the consumer-spawn doesn't?
4. `packages/mcp-server-kafka/src/services/client-manager.ts` — consumer factory; check whether consumers are reused across calls or always new
5. Look at how the MCP SDK transport surfaces the underlying request — the AgentCore proxy expects a specific path (likely `/health` or similar) to respond fast; if the consumer is blocking the event loop or main thread, no health endpoint can answer

### Fix directions to consider

**A. Fail-fast cap on consume time.** Even with `timeoutMs=5000` (5s, well under 30s) the tool times out the runtime. So the issue isn't the timeout window itself; something in the consumer construction is blocking longer than expected. Profile first.

**B. Background consumer pool.** Pre-create a consumer at process start, reuse for tool calls. Avoids per-call construction cost. Risk: ephemeral-group semantics break (every call would land in the same group; not what the tool's docstring promises).

**C. Pre-emptive health-check response.** If AgentCore's health endpoint is documented, ensure our MCP server responds to it before any tool work begins. The MCP SDK transport may already do this; verify.

**D. Convert to admin-only path.** Same as `kafka_get_topic_offsets` + a fetch-record API on the admin client (if `@platformatic/kafka` exposes one) — sidesteps the consumer entirely. Likely the cleanest fix if the API exists.

### Acceptance criteria (per the ticket)

* `kafka_consume_messages` either completes within the AgentCore health-check window OR fails fast with a clear "consume timeout exceeded" error that doesn't propagate to subsequent calls
* Calling `kafka_consume_messages` does NOT block subsequent tool calls (runtime stays healthy after return/error)
* `kafka_get_message_by_offset` (uses similar consumer-spawn machinery from SIO-679) is verified to NOT have the same problem — see SIO-700 below; these two need cross-validation

### Out of scope

* Rewriting the tool to not use a real consumer at all (the tool's value is reading actual message bytes; can't dodge that)
* Reducing the max `timeoutMs` below 5s — too aggressive for sparse-message production debugging

## SIO-700 — `kafka_get_message_by_offset` returns "Listing offsets failed"

### What we know

Live call against a real topic returns `MCP error -32603: Listing offsets failed`. The SIO-679 work introduced a discriminated-union return type (`ok | out_of_range | error`); we're hitting the `error` branch. The error itself is non-IAM-shaped (no `awsError` field) — its actual cause is hidden behind the generic string.

### Reproduction

```bash
# Wait ~30s after any kafka_consume_messages call (SIO-699 cooldown), then:
curl -sN -X POST http://localhost:3000/mcp \
  -H 'Accept: application/json, text/event-stream' \
  -H 'Content-Type: application/json' \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"kafka_get_message_by_offset","arguments":{"topic":"example-topic","partition":0,"offset":0}}}'
```

Returns:

```json
{
  "result": {
    "isError": true,
    "content": [{"type": "text", "text": "MCP error -32603: Listing offsets failed."}]
  }
}
```

`example-topic` is a real topic on `c72-shared-services-msk` (verified via `kafka_list_topics`). `kafka_describe_topic` and `kafka_get_topic_offsets` BOTH work against the same cluster (against `T_PRIVATE_SAP_CAR_PRICES` specifically), so the admin client's `listOffsets` RPC works in general — just not from this code path. Could be topic-specific.

### Three hypotheses (in priority order)

1. **Stale fixture** — `example-topic` may have offset 0 retention-deleted. The pre-validation calls `admin.listOffsets` for `(earliestOffset, latestOffset)`; if the topic has retention-deleted everything, the admin call may fail in a way the code reports as a generic "Listing offsets failed."
2. **Per-topic IAM gap** — the data-plane `kafka-cluster:DescribeTopic` permission might be resource-scoped and `example-topic` might not be covered. The probe showed `kafka_get_topic_offsets` works for `T_PRIVATE_SAP_CAR_PRICES` — try `T_PRIVATE_SAP_CAR_PRICES` with offset 0 to isolate.
3. **Generic error message swallows the cause** — SIO-679 added classification for 16 specific Kafka error codes (`OFFSET_OUT_OF_RANGE`, `UNSUPPORTED_VERSION`, etc.) but the `Listing offsets failed` path returns a non-classified `error` variant. The underlying `MultipleErrors` from `@platformatic/kafka` is being discarded BEFORE reaching the MCP response.

Hypothesis 3 is the most likely fix target — even if hypotheses 1 or 2 are also true, they'd be uncovered by surfacing the underlying error rather than the generic string.

### Where to start

1. **Test against a known-good topic first**: `T_PRIVATE_SAP_CAR_PRICES` partition 0 offset 0 (or whatever earliest offset `kafka_get_topic_offsets` reports for it). If that succeeds, hypothesis 1 is confirmed and the fix is mostly cosmetic. If it fails too, hypothesis 3 is the right path.

   ```bash
   # Get a known-good offset first
   curl -sN -X POST http://localhost:3000/mcp \
     -H 'Accept: application/json, text/event-stream' \
     -H 'Content-Type: application/json' \
     -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"kafka_get_topic_offsets","arguments":{"topic":"T_PRIVATE_SAP_CAR_PRICES"}}}' \
     | sed -n 's/^data: //p' | jq -r '.result.content[0].text' | jq '.partitions[0]'

   # Then call get_message_by_offset with that earliest offset
   ```

2. **Read `packages/mcp-server-kafka/src/services/kafka-service.ts`** — find `getMessageByOffset` (the SIO-679 commit was `ea6d0dc`). Look for the `Listing offsets failed` string. It will be in the generic-error branch BEFORE the SIO-679 classifier walks the `MultipleErrors` children.
3. **Mirror the SIO-679 BROKER_ERROR pattern** for this path: walk `error.cause` or the `MultipleErrors` structure, extract `kafkaErrorCode` + `kafkaErrorName`, return them in the structured `error` variant. The same 16-code classifier the existing BROKER_ERROR path uses should work here.

### Acceptance criteria (per the ticket)

* `kafka_get_message_by_offset` either:
  * Returns the actual message body when called on a topic with a valid offset, OR
  * Returns a structured error with the underlying broker error code (`OFFSET_OUT_OF_RANGE`, `UNKNOWN_TOPIC_OR_PARTITION`, etc.) — same shape SIO-679 already established for BROKER_ERROR
* Smoke test against a known-good topic+offset succeeds (validates hypothesis 1 either way)
* The generic `Listing offsets failed` string is replaced with classified errors in all paths

### Out of scope

* Re-architecting `getMessageByOffset` away from `@platformatic/kafka`
* Changing the SIO-679 discriminated union — keep the `ok | out_of_range | error` shape, just enrich the `error` variant

## Why these two are paired in one handoff

Three reasons:

1. **Same file, related code paths.** Both bugs live in `kafka-service.ts`. SIO-699's `consumeMessages` and SIO-700's `getMessageByOffset` both use `@platformatic/kafka` consumer-construction machinery. A fix to one may inform the other.
2. **Cross-validation requirement.** SIO-699's acceptance criteria explicitly says "verify `kafka_get_message_by_offset` does NOT have the same problem." If you fix SIO-699 by changing how consumer construction works, SIO-700's symptoms may shift. Worth knowing both ticket states at once.
3. **Same live setup cost.** Investigating either requires the AgentCore SigV4 proxy running, real MSK access, ~30s cooldowns between consume calls (because of SIO-699 itself). Pay the setup once, fix both.

Order suggestion: **SIO-700 first, then SIO-699.** SIO-700 is likely a small classifier-extension fix (mirror existing pattern), low risk, fast. Once it lands, you have a clean baseline for tackling SIO-699's larger architectural question (how to stop blocking the AgentCore health-check window), with SIO-700 as the cross-validation reference.

## Live testing setup

You'll need:

1. **AgentCore SigV4 proxy on port 3000.** The proxy is started outside the repo. Verify with:
   ```bash
   lsof -nP -iTCP:3000 -sTCP:LISTEN
   ```
   If not running, the user can start it (it's their tool, not a repo artifact).

2. **`.env` with the AgentCore credentials.** Already present at the repo root. Check:
   ```bash
   grep -E "^AGENTCORE|^KAFKA_MCP_URL" .env | head -5
   ```

3. **Cooldowns between calls.** SIO-699 itself causes `-32010` runtime-unhealthy responses for ~30s. Don't fire calls back-to-back during investigation — sleep 15-30s between probes if a previous one timed out.

4. **`jq` in PATH.** All probes use `jq` to extract content from the SSE response.

### Probe template

```bash
call() {
  local name=$1 args=$2
  resp=$(curl -sN --max-time 90 -X POST http://localhost:3000/mcp \
    -H 'Accept: application/json, text/event-stream' \
    -H 'Content-Type: application/json' \
    -d "{\"jsonrpc\":\"2.0\",\"id\":1,\"method\":\"tools/call\",\"params\":{\"name\":\"$name\",\"arguments\":$args}}")
  body=$(echo "$resp" | sed -n 's/^data: //p' | head -1)
  err=$(echo "$body" | jq -r '.error // empty | tostring')
  is_error=$(echo "$body" | jq -r '.result.isError // false')
  text=$(echo "$body" | jq -r '.result.content[0].text // ""')
  if [ -n "$err" ]; then echo "MCP_ERROR: $err"
  elif [ "$is_error" = "true" ]; then echo "TOOL_ERROR: $text"
  else
    awsErr=$(echo "$text" | jq -r '.awsError // empty' 2>/dev/null)
    if [ -n "$awsErr" ]; then echo "AWS_GAP: $awsErr"
    else echo "OK: $(echo "$text" | head -c 400)"
    fi
  fi
}

# Usage: call 'kafka_get_topic_offsets' '{"topic":"T_PRIVATE_SAP_CAR_PRICES"}'
```
