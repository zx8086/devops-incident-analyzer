## What was broken

Per the SIO-689 trace audit and the SIO-692 follow-up evidence: when a sub-agent throws (recursion limit, timeout, transient MCP error), the supervisor's alignment-retry path in `routeAfterAlignment` silently re-runs it on a fresh invocation. The first failure's evidence -- duration burned, error category, partial work -- is invisible to user-facing metrics because the second run's success masks it in the aggregate.

LangSmith trace `019e0d35-1e5b-7000-8000-02b04741fb6f` (the SIO-689 example) had two elastic-agent runs: a 202.6s recursion-limit failure followed by a silent 139.9s retry success. The eval scoreboard read "4/5 clean" while one query had actually consumed 342s of sub-agent latency.

Three downstream consequences (from the ticket):

1. **Eval signal degraded.** Pass-rate metrics treated "failed first try, retry succeeded" identically to "succeeded first try."
2. **Cost accounting wrong.** Bedrock token bills include both runs' LLM calls; without first-attempt tracking, cost was under-reported.
3. **Regression detection blind.** A change that bumped first-try failure rate from 5% to 25% wouldn't surface as long as retries still covered.

The retry mechanism itself is correct -- this was an observability gap, not a logic bug.

Four commits in the PR, each independently testable:

### 1. `summarizeFirstAttempts` helper (`packages/agent/src/alignment.ts`)

New exported function + `FirstAttemptSummary` interface. Takes the accumulated `dataSourceResults` array, returns one summary per datasource:

```ts
interface FirstAttemptSummary {
  dataSourceId: string;
  firstStatus: "success" | "error";
  firstCategory?: ToolErrorCategory;  // auth | session | transient | unknown
  firstDurationMs: number;
  recovered: boolean;  // first failed AND a later retry succeeded
}
```

Discriminates first attempts from retries via the existing `isAlignmentRetry` field on `DataSourceResult`. **No schema change.** For elastic deployment fan-out (multiple sibling first-attempt results), prefers a failing result over a successful sibling so the summary surfaces the failure that triggered retry. Categories come from `toolErrors[0].category` when present, falling back to `classifyToolError(error)` for catch-block failures.

6 unit tests in the new `packages/agent/src/alignment.test.ts` covering recovered, clean, non-recovered, tool-error-priority, dedup, and the recursion-limit-vs-transient distinction.

### 2. WARN at retry dispatch (`packages/agent/src/alignment.ts:209-222`)

Replaced the prior INFO log "Dispatching alignment retries" (which carried only `retryTargets`) with a WARN that includes `firstAttempts`:

```
!!! Dispatching alignment retries -- first-attempt failures masked by retry
{
  retryTargets: ["elastic"],
  skipped: [],
  retryAttempt: 1,
  firstAttempts: [
    { dataSourceId: "elastic", firstStatus: "error",
      firstCategory: "transient", firstDurationMs: 202609, recovered: true }
  ]
}
```

Promoted to WARN because masked failures are operationally significant for cost + regression detection. Fires only on retry-bearing runs -- the existing happy-path `info("Aggregating datasource results")` line stays untouched.

### 3. Conditional summary log in `aggregate` (`packages/agent/src/aggregator.ts`)

After the existing per-source aggregation summary, added:

```ts
if (firstAttemptFailureCount > 0) {
  logger.info(
    { firstAttemptFailureCount, recoveredCount, firstAttempts },
    "First-attempt sub-agent failures masked by retry",
  );
}
```

The conditional means clean runs stay quiet -- mirrors the SIO-693 boot-WARN design principle ("silent on the happy path"). The line shows up at the end of an eval run and lets a reader see "1 first-try failure recovered via retry" without scanning the whole tail.

### 4. LangSmith eval-output metadata (`packages/agent/src/eval/run-function.ts`)

`runAgent`'s return shape extended with `firstAttempts: FirstAttemptSummary[]` alongside `response`/`targetDataSources`/`confidenceCap`. Added at the same level as `response` (not nested), so the SIO-692 judge prompt's `output.response` short-circuit at `evaluators.ts:35` stays valid.

Field name `firstAttempts` matches the alignment WARN and aggregator info keys -- a reader cross-referencing log + LangSmith trace sees the same key everywhere.

### Test seams

Both `alignment.ts` and `aggregator.ts` got a small `_setAlignmentLoggerForTesting` / `_setAggregatorLoggerForTesting` seam (mirrors the SIO-693 `hasSeededTokensFn` pattern). Production callers go through the default pino logger -- the seam introduces one indirection level but is exercised in unit tests only. Avoids the `mock.module` brittleness for module-singleton loggers that the plan flagged as a risk.

## Why this design

- **No schema change.** Per scope choice ("core observability only" over the schema-change option). `summarizeFirstAttempts` derives the summary from existing `isAlignmentRetry`. Keeps `DataSourceResult` consumers (state reducer, aggregator results-block, frontend `datasource_progress` event) untouched. If a future ticket needs the field on the schema (e.g., per-result frontend display), it's an additive change, not a migration.
- **No new OpenTelemetry counter.** Per scope choice. The pino WARN/info already feeds OTel logs via `@devops-agent/observability`. A dedicated counter would require adding a Meter helper to `observability/src/otel.ts` (none exists today) and was deferred. If the log signal proves insufficient for dashboards, that's a follow-up.
- **No `eval:summary` post-eval script.** Per scope choice. The Task 4 LangSmith metadata makes per-query `firstAttempts` queryable from the LangSmith UI directly, which covers the acceptance criterion without needing a CLI walker.
- **`unknown` category for recursion-limit failures.** `classifyToolError`'s pattern set in `sub-agent.ts` doesn't include "recursion limit" -- such failures classify to `unknown`, which the alignment retry path treats as retryable (functionally equivalent to `transient`). Surfaced in test 1's documentation comment. Widening `classifyToolError` would be a behavior change beyond observability and was deferred. The category is preserved verbatim in the WARN so an operator can still tell apart a recursion failure from a 503 timeout.
- **Test seams over module-mocking.** `bun:test`'s `mock.module` is brittle for module-singleton loggers (the `getLogger("agent:alignment")` evaluates at import time). The seam is one exported function per module, off in production, and used only in unit tests.

## Gotchas (carry forward)

Mostly inherited. New gotchas from this session marked **NEW**.

1. **NEW: bun:test `mock.module` is brittle for module-singleton loggers.** `getLogger("agent:alignment")` evaluates at import time and caches the resulting logger as a module-scoped const. Mocking `@devops-agent/observability` after import doesn't replace it. The test-seam pattern (`_setAlignmentLoggerForTesting`, `_setAggregatorLoggerForTesting`) is the cleanest workaround -- mirrors SIO-693's `hasSeededTokensFn`. For any future test that needs to capture log emissions from a module-scoped logger, prefer a one-line exported test seam over `mock.module`.

2. **NEW: `classifyToolError` pattern set does not include "recursion limit".** Recursion-limit failures classify to `unknown`. The alignment retry path treats `unknown` as retryable, so behavior is correct, but the WARN/info `firstCategory` will show `unknown` for what is operationally a transient/recoverable failure. If you change this, also re-check the SIO-691 test "flags first-failed retry-succeeded as recovered" -- it expects `unknown` today.

3. **NEW: biome 2.4.12 disagrees with the user's "type imports before value imports" rule for files where the type import is from a longer-named module.** Specifically `import type { DataSourceResult } from "@devops-agent/shared"` should come AFTER `import { describe, ... } from "bun:test"` because biome sorts by source string and `@` < `b` lexicographically. The reference memory at `feedback_biome_type_before_value_imports.md` is generally true within a single source-name group, not across groups. When in doubt, run `bun run lint:fix` and accept the result.
