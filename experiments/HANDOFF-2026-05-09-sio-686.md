Pruning would amputate the ReAct loop's ability to refer back to earlier observations during correlation.
3. **Sizing approach:** instrument before we cap. Rather than picking a ceiling from intuition (64KB? 128KB?), ship instrumentation always-on, ship the cap behind an env flag default-off, then read the real distribution from LangSmith traces and tune in a follow-up.

The user chose the single-PR-with-flag shape over a two-PR sequence so the cap is hot-deployable per-environment without a second merge.

## Gotchas worth remembering

1. **The wrapper is a `Proxy`, not a class clone.** This matters because `StructuredToolInterface` extends `RunnableInterface` (it has `lc_namespace`, `bindTools`, callbacks, `withConfig`, etc. — many fields LangChain reads to decide tool-binding semantics). Reconstructing via `tool()` from `@langchain/core/tools` would change `responseFormat` handling for MCP tools (which use `"content_and_artifact"` per `node_modules/.bun/@langchain+mcp-adapters@*/dist/tools.js:458`). The Proxy passes through everything except `invoke`, so identity is preserved. **Don't "simplify" this to a class wrapper without re-confirming MCP tool result shapes still flow correctly through the ReAct loop.**

2. **`ToolMessage` rebuild preserves `tool_call_id`/`name`/`status`/`artifact`.** When the underlying tool returns a `ToolMessage` (which it does inside the ReAct loop, where LangGraph's `ToolNode` invokes with a `ToolCall` arg), the wrapper builds a new `ToolMessage` with the truncated content but copies these fields verbatim. Drop any of them and the LangChain message routing breaks (no tool_call_id → orphaned tool message; no status → error classification path in `extractToolErrors` at `sub-agent.ts:78` flips false-negative). Tested by the integration case `truncates oversized hits payload to first 3 hits when capBytes set`.

3. **`describeToolResult` records the *original* size, even when truncation fires.** This is intentional — we want LangSmith traces to show pre-truncation bytes so cap-tuning has accurate input. The `subagent.tool_result_truncated` event fires in *addition* to `subagent.tool_result`, never instead of it. Two events per truncated call.

4. **The cap is per-call, not cumulative.** A sub-agent invoking 30 tools with each result capped at 64KB can still produce 1.9MB of message history. That's fine — Bedrock's input window is much larger and the deep-analysis loop needs that depth. If we ever observe cumulative bloat causing the same symptom, that's a *different* fix (history pruning), not a tighter per-call cap. Keep them separate; user explicitly rejected pruning this session.

5. **`responseFormat: "content_and_artifact"` returns `[content, artifact]` when invoked with raw input** (not a ToolCall) and a `ToolMessage` when invoked with a ToolCall. The wrapper handles both: `extractContent()` checks for `"content" in result`, `rebuildResult()` instanceof-checks `ToolMessage` and falls through to a plain-object spread for shape-compatible non-ToolMessage results. The unit tests cover both paths.

6. **`getSubAgentToolCapBytes()` accepts an env override for testability.** The signature is `(env: NodeJS.ProcessEnv = process.env)`. In production it reads `process.env`; in tests it takes a fake `{ SUBAGENT_TOOL_RESULT_CAP_BYTES: "65536" }` so we don't have to mutate global state and risk cross-test pollution. **Do not** import this and call it without the override inside a test that runs alongside others — Bun's parallel test runner could leak the env mutation.

7. **`MAX_TOOLS_PER_AGENT = 25` (sub-agent.ts:101) still exists and is still useful.** It's the alignment-retry tier-2 fallback for the case where 84 tools confuse the model's tool-selection step. It's *not* a context-overflow safety net — that confusion was the original SIO-686 hypothesis-1 phrasing, but the actual root cause was always result size. Both mechanisms now coexist: tool-count cap protects tool-selection clarity, result-size cap protects context budget. They address different failure modes.

8. **Biome auto-formatting flagged the truncate-tool-output test on first commit.** Long inline type assertions get split across multiple lines under `bunx biome check --write`. If you add similar tests, write the type as multiline from the start to skip the autofix round-trip.

9. **`noUncheckedIndexedAccess` is on at the workspace level.** Destructuring `[wrapped] = instrumentTools(...)` types `wrapped` as `T | undefined`. The test file works around this with a `wrapOne()` helper that throws if the array is empty. Pattern lifted from `packages/agent/src/aggregator.test.ts`.

10. **The 200KB test fixtures matter.** The Q4 elastic failure trace estimated >75MB cumulative LangSmith trace size, so per-call payloads of 100KB+ are realistic. The unit-test fixtures use 200KB to ensure the cap path is forced regardless of how the payload compresses through `JSON.stringify`. If you tighten the test fixtures to "barely over the cap," you risk false-positives where the JSON-aware reducer produces ≤cap output but the test expected text-fallback.

## What NOT to do in next session

- **Don't flip `SUBAGENT_TOOL_RESULT_CAP_BYTES` on by default in code without first running the eval and reading p95 trace bytes.** That was the deliberate point of this PR's shape — pick the ceiling from data, not intuition.
- **Don't add history pruning across iterations to fix SIO-687.** User explicitly rejected this in this session; deep-analysis correlation needs the full message history. If 687 turns out to need a focused fix, reach for `langsmith.ts:14` (`trimLargeValues`) tuning, batched submission, or per-tool soft caps at the MCP server, not history pruning at the agent.
- **Don't switch the Proxy wrapper to a class-based clone "for clarity"** — it would change `responseFormat` handling and require re-validating every MCP tool's result shape. See gotcha 1.
- **Don't drop the `_truncated`/`_totalHits`/`_totalCount`/`_nodeCount` markers** when extending the JSON-aware reducer — the model uses them to recognise that the payload was reduced and avoid asserting based on incomplete data. Removing them silently regresses correlation quality.
- **Don't mark SIO-686 as Done in Linear without explicit user approval** (per CLAUDE.md and the standing rule from prior sessions). It's currently In Review with PR #43 attached.
- **Don't bundle SIO-687 work into a "while we're here" PR** with the cap-tuning follow-up. Same reasoning the prior handoff used for keeping 685/686/687 separate: each ticket has hypotheses to investigate; bundling skips the investigation.
