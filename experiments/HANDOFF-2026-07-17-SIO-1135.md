## TL;DR

SIO-1134 (PR #398) established the strict retention model: an investigation only counts as durable memory once a human curates it by linking a Jira ticket (`Incident.ticketKey`). Enrichment already filters to curated-only, but two gaps remain: (1) uncurated Incident rows are never physically removed, and (2) the `kg-incident` / `kg-root-cause` mirror facts are still written on EVERY run, so rebuild-from-facts resurrects uncurated incidents forever (agent-memory facts are immutable). Success = a retention sweep deletes old uncurated incidents, mirror facts are written only at curation time, and a rebuild from facts produces only curated records.

## Context -- how this ticket came to be

The user's decision (2026-07-16, via AskUserQuestion): "Only ticketed runs persist" -- if an investigation did not yield the right analysis, it never gets a Jira ticket, and it must not pollute memory. SIO-1134 implemented the visibility half (see `docs/superpowers/specs/2026-07-16-hil-learning-loop-design.md` for the lane design, and PR #398 for the curation signals). This ticket is the cleanup half, explicitly split out during SIO-1134 review.

## Where the bodies are buried

1. **Per-run incident row + fact write** -- `packages/agent/src/graph-knowledge.ts:95-120` (`recordGraphEntities`): every complex-query turn MERGEs an `Incident {id: state.requestId}` AND writes a `kind:"kg-incident"` fact via `recordKeyDecision`. The fact is the problem: immutable, replayed by rebuild.

2. **Per-run root-cause fact** -- grep `kind: "kg-root-cause"` in `packages/agent/src/` (main pipeline writer records it when a correlation rule wins; the HIL path in `learn/apply.ts:158-171` also writes one, but that one is fine -- it only fires on human approval).

3. **Curation-time fact already exists as the model to follow** -- `packages/agent/src/learn/apply.ts:197-225`:

   and `apps/web/src/routes/api/tickets/[provider]/+server.ts:16-35` (`curateIncident`, fired from POST when `body.requestId && created.key`).

4. **Rebuild replay order** -- `packages/knowledge-graph/src/rebuild.ts:179-182`:


## The fix (step-by-step)

1. **New writer** `purgeUncuratedIncidents(store, cutoffIso)` in `packages/knowledge-graph/src/writer.ts`. lbug 0.14.3 = single-clause Cypher only: delete edges first, then nodes, in separate statements, e.g.
   - `MATCH (i:Incident)-[r:AFFECTED_BY]->() WHERE i.ticketKey = '' AND i.createdAt < $cutoff DELETE r` (repeat per edge type: `AFFECTED_BY`, `HAS_ROOT_CAUSE`, `RESOLVED_BY`, `DISCOVERED_DURING`, plus any incident-touching rel added since -- grep `FROM Incident|TO Incident` in schema.ts)
   - then `MATCH (i:Incident) WHERE i.ticketKey = '' AND i.createdAt < $cutoff DELETE i`
   - Return counts. Risk: deleting rows from the vector-indexed table -- validate against the real engine in `ladybug.integration.test.ts` (in-memory fake cannot).
2. **Retention config**: `KG_UNCURATED_RETENTION_DAYS` (default ON, e.g. 30 -- user rule: features default ON). Read at call time, injectable env param (no module-scope Bun.env in packages/agent).
3. **Wire the sweep** beside the topology cron in `apps/web/src/lib/server/agent.ts` (same install seam as `runTopologySweep`); soft-failing, logged, never blocks a turn.
4. **Move `kg-incident` fact to curation time**: delete the `recordKeyDecision` call in `recordGraphEntities`; instead, both curation sites (`curateIncident` endpoint + `applyLearnings`) write the `kg-incident` fact by reading the CURRENT graph row (summary/severity/services) so annotations match `incidentFromAnnotations` in rebuild.ts:96-ish exactly. Keep `kg-incident-ticket` as-is.
5. **Move/gate `kg-root-cause` fact**: main-pipeline machine root causes should only become facts for curated incidents. Simplest correct order: keep the graph write per-run, write the fact at curation time by reading `rootCauseForIncident` (reader exists, used in `learn/match.ts`). The HIL apply-path fact write stays (already curation-gated).
6. **Rebuild parity test** in `packages/knowledge-graph/src/rebuild.test.ts` (or the existing rebuild coverage): replaying a fact set containing only curated writes yields only curated incidents; uncurated ids absent.

## Risks and edge cases

| Risk | Likelihood | Mitigation |
|---|---|---|
| Purge deletes an incident the user was about to curate via `learn from` | Medium | retention window (default 30d) + purge only `ticketKey = ''`; legacy reconciliation happens via the learn-lane fallback (see SIO-1134 notes) |
| Vector-indexed table delete misbehaves in lbug | Low | real-engine integration test before merge |
| Fact annotations drift from rebuild mappers | Medium | copy shapes from `rebuild.ts` mappers verbatim; add parity test |
| Web app imports knowledge-graph directly | Certain if attempted | apps/web has NO direct knowledge-graph dep (isolated linker escapes to stale code); route via `@devops-agent/agent` re-exports (`packages/agent/src/index.ts:3-5`) |
