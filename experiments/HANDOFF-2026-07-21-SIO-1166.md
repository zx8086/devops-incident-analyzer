## TL;DR

SIO-1163 (merged, PR #430) added WAL-corruption auto-recovery to `LadybugStore.openDatabase()`: on `new lbug.Database(path)` throwing a message matching `/corrupted wal file/i`, it quarantines the `.wal` file and retries construction once. Direct testing today against the REAL corrupted WAL from the SIO-1165 incident (safe scratch copies, never touching production data) revealed this catch may not reliably fire at all: **lbug's `Database` construction is lazy** (its own docstring says the file isn't opened until the first query), so the actual WAL replay can be deferred past the constructor to the first `conn.query()` call -- which is *outside* `openDatabase()`'s try/catch entirely. Five-plus repeated trials against byte-identical copies of the same corrupted file gave inconsistent results: sometimes the constructor threw as SIO-1163 assumes, sometimes it succeeded cleanly and the failure only appeared later as a raw uncaught `SIGABRT`, sometimes a full `init()` DDL loop segfaulted Bun outright. Success for this ticket = a real understanding of exactly which lbug call surfaces a torn-WAL failure and reliable code that catches it there, backed by a deliberately-crafted-corrupt-WAL test fixture (not a mocked module, not scratch copies of production data).

## Context -- how this ticket came to be

Timeline, all 2026-07-20/21:
1. **SIO-1163** (2026-07-20): recurring "Corrupted wal file. Read out invalid WAL record type." warnings across three cron/route call sites, caused by `getGraphStore()`'s singleton caching a permanently-rejected promise. Fixed: `openDatabase()` catches that specific message, quarantines the WAL, retries; `getGraphStore()` resets its singleton on any rejection instead of latching forever. Shipped with a `store.test.ts` suite that mocks the `lbug` module via an injectable `_setLbugLoaderForTesting` seam -- the fake `Database` constructor throws synchronously by design, which validates the CODE's control flow (quarantine-rename-retry-rethrow) but never proved the REAL native engine's constructor actually throws for genuine WAL corruption.
2. **SIO-1165** (2026-07-21, next day): a NEW error signature appeared in production -- `KU_UNREACHABLE` (a native C++ assertion in lbug's storage layer, `types.cpp:322`). Investigation (reading lbug's own bundled source at `packages/knowledge-graph/node_modules/lbug/lbug-source/`) proved this is the SAME root cause as SIO-1163 (a torn WAL tail from a crash mid-write) manifesting through a different code path: a record whose declared length runs past EOF triggers `BufferedFileReader`'s "Reading past the end of the file" instead of the `INVALID_RECORD`/"Corrupted wal file" message SIO-1163's regex matches. Fixed the live incident by manually quarantining the WAL (same recipe as SIO-1129) and independently verifying the base file alone was clean via direct query (743 nodes, repeated trials).
3. **This ticket (SIO-1166)**, filed same day as a natural follow-up: "should we broaden `WAL_CORRUPTION_PATTERN` to also catch the 'reading past end' text?" Testing that question directly against the real corrupted WAL is what surfaced the deeper problem -- the catch may be in the wrong place in the call stack, not just matching the wrong text.

## Where the bodies are buried

**The current (shipped, on `main`) detection + retry logic** -- `packages/knowledge-graph/src/store.ts:60,109-147`:

```ts
const WAL_CORRUPTION_PATTERN = /corrupted wal file/i;

export class LadybugStore implements GraphStore {
	private db: LbugDatabase | null = null;
	private conn: LbugConnection | null = null;

	constructor(private readonly path: string) {}

	private async connection(): Promise<LbugConnection> {
		if (this.conn) return this.conn;
		const dir = dirname(this.path);
		if (dir && !existsSync(dir)) mkdirSync(dir, { recursive: true });
		const lbug = await loadLbug();
		this.db = await this.openDatabase(lbug);       // <-- ONLY this call is guarded
		this.conn = new lbug.Connection(this.db);        // <-- NOT guarded
		return this.conn;
	}

	private async openDatabase(lbug: LbugModule): Promise<LbugDatabase> {
		try {
			return new lbug.Database(this.path);           // <-- the ENTIRE guarded surface
		} catch (error) {
			const message = error instanceof Error ? error.message : String(error);
			if (!WAL_CORRUPTION_PATTERN.test(message)) throw error;
			const walPath = `${this.path}.wal`;
			if (existsSync(walPath)) {
				const quarantinePath = `${walPath}.corrupt-${Date.now()}`;
				renameSync(walPath, quarantinePath);
				logger.warn(/* ... */);
			}
			return new lbug.Database(this.path);           // <-- retry, same process, same call
		}
	}

	async init(): Promise<void> {
		const conn = await this.connection();
		for (const ddl of MIGRATIONS) {
			await conn.query(ddl);                          // <-- NOT guarded -- this is likely
		}                                                   //     where replay ACTUALLY fails
		// ... ALTER_MIGRATIONS, VECTOR_INDEX_SETUP loops, also unguarded ...
	}
```

**Why this matters**: `packages/knowledge-graph/node_modules/lbug/database.js`'s own JSDoc on the `Database` constructor: *"Note that the initialization is done lazily, so the database file is not opened until the first query is executed."* The WAL replay is part of that lazy open. If the replay doesn't run until `conn.query(ddl)` inside `init()`, then `openDatabase()`'s try/catch (which only wraps `new lbug.Database(this.path)`) can never see the error -- it surfaces three call-frames later, completely unguarded.

**Today's reproduction results** (all against safe scratch copies of the real corrupted WAL from SIO-1165 -- copied from `apps/web/.data/knowledge-graph` + `apps/web/.data/knowledge-graph.wal.corrupt-2026-07-21`, NEVER touching the live files):

| Trial shape | Result |
|---|---|
| `new lbug.Database(path)` alone, fresh copy, repeated 5x | ALL 5 succeeded with no error (construction is lazy -- doesn't trigger replay) |
| `new lbug.Database(path)` -> `new lbug.Connection(db)` -> `conn.query("MATCH (n) RETURN count(n)")` | Constructor + Connection succeeded, `query()` crashed with raw `SIGABRT` (exit 134) -- NOT a catchable JS rejection |
| Full `LadybugStore.init()` (real MIGRATIONS/ALTER_MIGRATIONS/VECTOR_INDEX_SETUP DDL loop) against the SAME corrupted WAL, quarantine-and-retry already having run inside `openDatabase()` | Bun segfaulted (`panic: Segmentation fault`), before any output printed |
| Base file ALONE (WAL entirely absent from the start, never present) + full `init()` + query | Clean every time -- 743 nodes, no crash, repeated across many trials |
| `throwOnWalReplayFailure=false` (lbug's own built-in tolerant-replay flag, 8th positional ctor arg, undocumented in our code) against the corrupted WAL | Construction succeeded; first query STILL segfaulted |

Earlier in this same investigation session (documented in Linear + memory), one trial DID observe `new lbug.Database(path)` throwing synchronously with "Reading past the end of the file ... .wal ... " -- so it's not that the constructor NEVER throws, it's that it's **non-deterministic** whether replay happens at construction or is deferred. This is the crux of what needs characterizing.

**The existing test suite that didn't catch this** -- `packages/knowledge-graph/src/store.test.ts` (169 lines, added in SIO-1163, extended by CodeRabbit-driven fixes same day). The `describe("LadybugStore WAL-corruption recovery", ...)` block uses `_setLbugLoaderForTesting(mockLbugLoader(...))` where `mockLbugLoader` is a hand-rolled fake whose `Database` class constructor throws synchronously exactly when told to (`packages/knowledge-graph/src/store.test.ts:19-44`). This is a valid test of "does the code correctly quarantine-and-retry WHEN the constructor throws" -- it says nothing about whether the REAL native engine's constructor throws at the right time for real corruption.

## The fix (step-by-step) -- NOT YET DESIGNED, this is where to start

This ticket is investigation-first, not a known fix applied mechanically. Suggested approach:

1. **Build a real (not mocked) corrupt-WAL test fixture.** Do NOT use scratch copies of production data (what today's investigation did, ad hoc, cleaning up after itself each time -- fine for one-off diagnosis, not repeatable/committable). Instead:
   - Programmatically create a tiny fresh lbug store (few nodes), let it write a WAL, then truncate/corrupt the `.wal` file at a controlled byte offset (e.g. mid-record) to deterministically reproduce EITHER the `INVALID_RECORD` variant or the "reading past end" variant on demand.
   - This needs the real `lbug` package installed (`packages/knowledge-graph/node_modules/lbug` -- check if it's still there via `ls packages/knowledge-graph/node_modules/lbug` before starting; it was present as of 2026-07-21 for this investigation, at version pinned in `packages/knowledge-graph/package.json`'s `"lbug": "^0.14.3"`).
   - Isolate this fixture-creation script under `packages/knowledge-graph/` so bare `import("lbug")` resolves (a script outside that package tree fails to find the module even with the right cwd -- confirmed today, costly 20 minutes of confusion).

2. **Characterize exactly where replay happens.** Run the fixture-corruption scenario N times (suggest N=20+, given today's observed nondeterminism) and record: does `new Database()` throw? Does it throw the SAME message every time for the SAME corruption pattern? Does `conn.query()` throw/segfault/succeed? Does a full `init()` DDL loop behave differently than a single simple query?

3. **Based on characterization, likely fix shape**: move (or duplicate) the WAL-corruption catch to wrap the FIRST real query/DDL statement after opening -- i.e., inside `LadybugStore.init()`'s first `conn.query(ddl)` call (or wrap the whole `init()` body), not just the `Database` constructor in `openDatabase()`. This might mean:
   - Catching around the first iteration of the `MIGRATIONS` loop in `init()` (`store.ts:151-153`), detecting the same WAL-corruption signatures there, and if caught, quarantining the WAL and returning a signal that forces the CALLER (`getGraphStore()`) to reconstruct a brand-new `LadybugStore` against the clean file -- NOT retrying within the same `LadybugStore` instance/native `Database` handle, since today's testing also showed that retrying construction on the SAME native handle inside the SAME process call can itself segfault on the subsequent `init()`/query (see trial 3 in the table above -- quarantine-and-retry succeeded at the `Database()` level but then STILL segfaulted on the DDL loop that followed in the same call).
   - This likely means: on WAL-corruption detection (wherever it's actually caught), the SAFEST recovery is not "retry in this call" but "quarantine the WAL, throw/reject cleanly, let `getGraphStore()`'s existing singleton-reset (already shipped, `store.ts:267-279`) force a genuinely fresh `getGraphStore()` call next time -- which today's testing showed IS safe (base file alone, freshly opened, never crashes)." The risk is entirely in the same-call retry-after-failure sequence, not in reopening from scratch afterward.

4. **Decide whether `openDatabase()`'s current retry-in-`openDatabase()` should be REMOVED** rather than extended, given point 3 -- i.e., catch-quarantine-rethrow (no retry) may be strictly safer than catch-quarantine-retry, once the catch is moved to the right place.

5. Write real-engine integration tests in `packages/knowledge-graph/src/ladybug.integration.test.ts` (the existing pattern for real-lbug-engine tests, `describe.skipIf(!lbugLoadable())`) using the fixture from step 1, asserting the new catch point actually engages and the resulting recovered store is queryable without segfaulting -- this is the test SIO-1163's mocked suite could not provide.

## Risks and edge cases

| Risk | Likelihood | Mitigation |
|---|---|---|
| Fixture-based corruption doesn't reproduce the exact nondeterminism seen against real production WAL | Medium | run many trials (20+) against the fixture; if it's ALWAYS deterministic on the fixture but wasn't against production, the nondeterminism may be file-size/memory-layout dependent -- worth testing at multiple fixture sizes |
| A "fix" that only moves the catch but still retries in-process could reintroduce today's segfault-after-retry finding | Medium-High | test the FULL sequence (catch -> quarantine -> whatever recovery -> subsequent init()/query) end to end, not just the catch point in isolation |
| lbug native module absent on CI/some dev machines | Certain, by design | real-engine tests already skip via `lbugLoadable()` guard (see `ladybug.integration.test.ts` pattern) -- new tests must follow the same skip-if-absent convention |
| Bun's own finalizer segfault (SIO-954, unrelated pre-existing issue) muddies test results | Medium | diagnose via package-level `bun run test`, trust a printed pass summary over a raw exit code; see `reference_lbug_cypher_and_teardown_gotchas` memory |
| Scope creep into "just disable WAL entirely" or similar drastic changes | Low but flagged | out of scope -- WAL is core to lbug's durability model; the fix should be about correct detection/recovery, not disabling durability |
