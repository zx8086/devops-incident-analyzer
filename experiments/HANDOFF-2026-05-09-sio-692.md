## SIO-692: judge stuck at 0.0 (PR #49)

### What was broken

The `response_quality` LLM-as-judge in the LangSmith eval suite scored 0.0 across every query in the most recent eval (`agent-eval-sio690-9d1cf3e-ae67f9fa`, post-PR-#47), making the eval signal load-bearing for ship-readiness gating useless. Earlier runs (SIO-688/SIO-689) had produced a 0.05–0.74 spread; the regression appeared post-`ae806d3` (the SIO-680/SIO-682 MCP-init fix).

### Investigation (the part not to repeat)

Pulled per-query `comment` strings via `langsmith run list --project agent-eval-sio690-9d1cf3e-ae67f9fa --name runAgent --full --format json`. All 5 comments were **substantive** — full sentences explaining what the judge thought was missing. This ruled out:

- (a) Parsing bug — would have yielded the literal string `"empty response"` from the `JSON.parse(... ?? '{...}')` fallback at `evaluators.ts:50`. None of the 5 said that.
- The "missing rubric or response" short-circuit at `evaluators.ts:35` — would have yielded that exact string. None did.

Leaving (b) stale rubric or (c) strict judge.

The actual root cause turned out to be **two compounding issues**, not one:

1. **Trajectory-shaped rubrics.** Rubrics named tool methods verbatim (`"Should call atlassian.runbook_lookup AND incident_correlation"`) and asked about which deployment was queried — but the judge only receives `run.outputs.output.response` (the final synthesized AI message at `run-function.ts:40-44`). It has zero visibility into tool calls.
2. **Strict judge prompt.** gpt-4o-mini at temperature 0 with a one-line system prompt was reading rubric clauses literally and missing semantically equivalent content. Q2's response cited GitLab commits febf6b28, 27cb9ce2, 10b78c86 with merge dates; the judge's comment said "does not reference recent GitLab deploys."

**Verified compounded — the rubric rewrite alone was insufficient.** Rewrote rubrics first, ran an intermediate eval (`agent-eval-15479db-sio692-651e7314`): still flat at 0.0 / 0.0. The judge was still being literal. Hardened the judge system prompt with per-clause reasoning + semantic-equivalence guidance + "concrete evidence counts" framing. Re-ran (`agent-eval-15479db-sio692-r2-8b6a9eec`): **0.2 / 0.4**.

## SIO-687: NOT closeable as a no-op

The SIO-693 handoff's "smallest item — likely a no-op closure if no 422s recur" was **wrong**. Today's r2 eval surfaced 2 fresh 422s.

### Evidence

| Run | Experiment | 422 count | Sizes |
|---|---|---|---|
| r1 | `agent-eval-15479db-sio692-651e7314` | 0 | — |
| r2 | `agent-eval-15479db-sio692-r2-8b6a9eec` | 2 | 27,751,411 bytes (27MB) and 75,364,219 bytes (75MB) |

Same agent code, same dataset, same MCP servers between r1 and r2 — the 422s are **non-deterministic** across runs. That points to payload-content variance (sub-agent message-history accumulation depends on tool-retry counts, which vary), not a deterministic missing-patch path.

### What's confirmed not the cause

The three patches in `packages/agent/src/langsmith.ts` (`patchClient` for CJS, ESM, and RunTree.sharedClient) ARE applied on every run — confirmed by three `Patched ... LangSmith client payload limits` log lines in both r1 and r2 logs. They just aren't sufficient.

### What is the cause (hypothesis)

The 422 message format is `patch.<run-id>.outputs: field size 75364219 exceeds maximum`. The `patch.` prefix is significant — it's an UPDATE to an existing run, not the initial CREATE. The `processInputs`/`processOutputs` overrides in `langsmith.ts:48-49` likely apply on the create path; the patch path may bypass them. Worth tracing `langsmith` SDK's PATCH-multipart code path to confirm where (and whether) `processOutputs` is invoked there.

### Recommendation

Don't close. Either:

- (a) Pair with SIO-691 fix — both stem from sub-agent message-history accumulation; fixing the upstream source could resolve both.
- (b) Add per-field truncation on the langsmith SDK's patch-multipart path specifically.

## Gotchas (carry forward)

Mostly inherited. New gotchas from this session marked **NEW**.

1. **NEW: langsmith CLI has no dataset upsert.** `langsmith dataset upload` is create-only; `langsmith dataset create` is empty-only. Updating dataset content requires `langsmith dataset delete --yes` then re-upload. Past experiments retain their inline rubric snapshots, so historical results stay valid against the OLD rubrics, not against the new ones. Plan for the `reference_dataset_id` to change on every dataset edit.

2. **NEW: gpt-4o-mini at temperature 0 with a one-line system prompt is systematically literal.** It will say "does not reference X" against responses that obviously reference X with concrete evidence (commit SHAs, timestamps, etc.). For LLM-as-judge use cases, the system prompt MUST include semantic-equivalence guidance + "concrete evidence counts" framing, or accept that scores will be flat 0. The fix in `evaluators.ts` (PR #49) is the template.

3. **NEW: rubric rewrites alone may be insufficient when paired with a strict judge.** The hypothesis "trajectory-shaped rubrics → strict-judge fails" was correct but incomplete — even response-content rubrics fail under a literal judge. Verify with one intermediate eval before assuming the rubric rewrite is sufficient. Cost is one extra eval run (~$0.50–1.50, ~14 min), saves a wrong-direction PR.

4. **NEW: 422s in eval traces are non-deterministic across structurally-identical runs.** r1 had 0, r2 had 2 with identical agent code. Sub-agent retry counts (and the resulting message history accumulation) vary between runs — same upstream cause as SIO-691.

5. **Run eval scripts from workspace root.** Bun's `.env` auto-discovery doesn't walk up from a sub-package cwd. Use `bun --env-file .env packages/agent/src/eval/run-eval.ts`. (Inherited.)

6. **NEW (re-verified): localhost precheck blocks AgentCore-hosted MCPs.** The `precheck.ts` spawn from `run-eval.ts` uses a relative path that fails when invoked from repo root (`Module not found "src/eval/precheck.ts"`). Konnect being AgentCore-only means precheck would fail even if the path resolved. (Inherited; re-verified.)

7. **Bun process hangs post-`Done.` on LangSmith trace upload.** Safe to `kill <pid>` after `Done.` prints. (Inherited.)
