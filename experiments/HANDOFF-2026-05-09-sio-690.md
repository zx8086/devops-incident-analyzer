## What was broken

Per the SIO-689 handoff trace categorization: during the failing pass-4 elastic sub-agent run, call #6 was an `elasticsearch_get_mappings` invocation that errored at exactly **60013 ms** with an empty payload, burning two LangGraph recursion steps for zero signal. Sibling cat-style discovery calls (`indices_summary`, `list_indices`) completed in ~4 s on the same trace, so the cause was specific to how `get_mappings` interacts with the shared client config.

Root cause confirmed in code:

- `packages/mcp-server-elastic/src/server.ts:45-46` builds the `Client` with shared `requestTimeout: 30000` + `maxRetries: 3` from `src/config/defaults.ts:28-29`.
- The `@elastic/elasticsearch` client applies `requestTimeout` **per attempt**, not total. With one transparent retry on a transient pool failure (e.g. dead-connection re-route), the wall-clock is `30 s + ~30 s ≈ 60 s` -- exactly the observed 60013 ms.
- `core/get_mappings.ts:52` invoked `esClient.indices.getMapping({ index })` with no per-request `TransportRequestOptions`, inheriting the cluster-wide policy intended for search/aggregation calls.

### Error mapping

`get_mappings.ts` adds a `timeout` branch in its error helper that maps to `ErrorCode.RequestTimeout` with an actionable hint ("narrow the index pattern to a single backing index, or skip this tool and infer fields from a sampled search"). Other discovery tools either already had a `timeout` branch (`indices_summary`, etc.) or surface the timeout through their generic execution branch -- the SDK message is preserved either way so the LLM agent can route around the failed call. Documented in the helper file's docstring.

### Why this design

- **Per-request override beats client-wide change.** The 30s/3-retry policy is correct for search and write paths; only discovery-class calls need fast-fail. The shared client constructor in `server.ts:33-89` was deliberately left alone.
- **`maxRetries: 0`, not `1`.** A 60s symptom from one retry already proves retries on discovery calls are net-negative for the agent -- an immediate fast-fail the LLM can route around beats a silent ~60s wait.
- **`8000` ms default.** `indices_summary` and `list_indices` complete in ~4 s on the same `eu-b2b` cluster (per SIO-689 trace); 8 s gives 2× headroom for tail-latency without approaching the recursion-eating regime.
- **Single helper.** Future tool authors get an obvious answer to "should I use this?" via the helper's docstring (which also enumerates the discovery scope and excludes search/bulk/write paths explicitly).

### Out of scope (not pulled in)

- Did not change `defaultConfig.elasticsearch.requestTimeout` or `maxRetries` (per ticket).
- Did not add retry-on-timeout (per ticket -- would re-burn the budget).
- Did not lower the SIO-689 recursion limit back to 25.
- Did not touch search/bulk/write tools.

## Gotchas (carry forward, mostly inherited from SIO-686/688/689 sessions)

1. **Run eval scripts from workspace root.** Bun's `.env` auto-discovery doesn't walk up from a sub-package cwd. Use `bun --env-file .env packages/agent/src/eval/run-eval.ts`.

2. **Localhost precheck blocks AgentCore-hosted MCPs.** Konnect is AgentCore-only and not bound locally; precheck would fail. Recreate `run-eval-no-precheck.ts` in `packages/agent/src/eval/` from the `run-eval.ts` template (drop the `spawnSync` precheck call); **do NOT commit it**. This session created+used+deleted it.

3. **Bun process hangs post-`Done.` on LangSmith trace upload.** Safe to `kill <pid>` after `Done.` prints -- experiment data is already submitted. Verified again this session.

4. **`tail -N` buffers stdout.** Pipe straight to a file (`> /tmp/run.log 2>&1`) or use `Monitor` with `tail -f | grep --line-buffered`.

5. **Confidence scores are LLM-judge stochastic.** Single-run deltas are noise; multi-pass averages are signal. (Not relevant to SIO-690's binary scoring but inherited as a load-bearing convention.)

6. **`MCP_OAUTH_HEADLESS=true` lives in the MCP server process, not the eval process.** If gitlab/atlassian popups appear during a "headless" eval, restart the MCP server (or `bun run dev`) with the flag in *its* env. **This session learned this the hard way -- see SIO-693.** `.env` was edited to set it permanently in this session.

7. **`bun run dev` runs `--filter '*' dev` across all workspace packages.** Some packages bind ports (couchbase :9082, gitlab :9084, atlassian :9085, elastic :9080), others bind to non-default ports (kafka via AgentCore proxy on :3000), konnect doesn't bind locally at all. If a port is missing from `lsof -i :908x`, check whether the package is meant to be local at all before assuming a crash.

8. **`langsmith experiment get` returns aggregate only.** For per-query feedback comments, use `langsmith run list --project <experiment-name> --name runAgent --full --format json` and parse `feedback_stats` per-run.

9. **Trace categorization for sub-agent runs requires the langsmith Python SDK,** not just `langsmith-fetch` CLI. The CLI's `traces` command pulls top-level eval-judge runs; child runs (where the actual sub-agent error and tool-call breakdown live) need `client.list_runs(project_id=..., trace_id=...)` paginated. Use the venv at `/Users/Simon.Owusu@Tommy.com/.local/pipx/venvs/langsmith-fetch/bin/python` -- it already has the SDK. (Inherited from SIO-689 followups.)
