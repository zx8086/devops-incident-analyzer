## What was broken

LangSmith trace upload returned `Failed to send multipart request. Received status [422]: ... field size N exceeds maximum allowed size of 26214400 bytes` on `patch.<run-id>.outputs` in some eval runs. SIO-693's closeout suggested it might be a no-op closure if 422s didn't recur. SIO-692's r2 eval had since shown 2 fresh 422s, and the auto-memory note documented the bug as non-deterministic across runs (~33% rate).

The pre-existing `packages/agent/src/langsmith.ts` patched three Client singletons (CJS, ESM, RunTree) via a single `require.resolve("@langchain/core/singletons")`. The patch demonstrably worked on the resolved Client (proven by `Patched CJS/ESM/RunTree.sharedClient LangSmith client payload limits` log lines on every eval startup). But 422s persisted.

---

## What we found (the diagnostic)

The bun module graph contains **multiple resolved copies of `@langchain/core`** because of peer-dep version skew:

- Top-level: `@langchain/core@1.1.40` (workspace dep) -- the only one the old patch covered.
- `@langchain/aws@1.3.4` and `1.3.5` peer-depend on `@langchain/core@^1.1.39`, which Bun resolves into a **separate `@langchain/core@1.1.39` copy** at `node_modules/.bun/@langchain+core@1.1.39/...`. This copy's `getDefaultLangChainClientSingleton()` returns a distinct unpatched Client instance.

Same applies to `langsmith` itself: `langsmith@0.6.3` (top-level) vs `langsmith@0.5.12` (nested under the older `@langchain/core` copy). Each has its own `RunTree.sharedClient`.

**Captured evidence** from eval r3 (2026-05-10, pre-fix):

- `LANGSMITH_FAILED_TRACES_DIR=/tmp/sio687-fail-dump` -- the SDK already has built-in capture of rejected multipart bodies at `langsmith@0.6.3/client.js:1782-1786`. Setting the env var triggered it on the next 422.
- The captured body: 75,368,320 total bytes, single field `patch.<run-id>.outputs` at 75,365,665 bytes. Decoding revealed:
  - Outputs dict had exactly ONE key `output` holding 75,365,709 bytes.
  - Content shape: a serialized LangChain ToolMessage (`{lc_serializable: true, lc_kwargs: {type: "tool", tool_call_id, status, content}}`).
  - The tool call was `elasticsearch_get_index_mappings` returning a multi-MB index mapping.
  - **No `[truncated: N bytes]` markers anywhere** -- proving `processOutputs` never ran on the Client that emitted this upload.

Cross-referencing the auto-memory hypothesis "patch-multipart path bypasses processOutputs": **wrong**. Reading `langsmith@0.6.3/dist/client.js`:

- `multipartIngestRuns()` at line 1522 handles both POST and PATCH.
- `prepareRunCreateOrUpdateInputs()` at line 937 calls `processInputs(...)` (line 940) and `processOutputs(...)` (line 943) for both runCreates AND runUpdates.

So `processOutputs` does fire on the patch path -- but only on the Client we patched. The 1.1.39 Bedrock-pipeline Client never gets touched.

## What shipped

### `packages/agent/src/langsmith.ts` (rewrite)

The old `patchClientPayloadLimits` did three hardcoded `require`/`import` calls. The new version walks `node_modules/.bun/` for `@langchain+core@*` and `langsmith@*` directories, dynamically imports each `dist/singletons/tracer.{cjs,js}` and `dist/run_trees.{cjs,js}`, and patches every singleton found. Forward-compatible: any future-added `@langchain/core` peer-dep variant gets caught automatically.

Key changes:

**Patch strategy: monkey-patch `Client.prototype` on every `langsmith` copy.** Initially attempted instance-level patching (every singleton + RunTree.sharedClient), but verification eval v2 reproduced the 422 with an identical 75MB fingerprint despite all 8 known instances being patched. The investigation showed `LangChainTracer.getRunTreeWithTracingConfig` (`@langchain/core/dist/tracers/tracer_langchain.js:90-97`) constructs new `RunTree` instances with `client: this.client`, and `tracer_langchain.js:82` reassigns `this.client = runTree.client ?? this.client` when a runTree arrives via async-local-storage. Tool-execution paths in LangGraph hit this and bypass our singleton patches.

Pivoted to **patching `Client.prototype.processInputs` and `Client.prototype.processOutputs` directly** — affects every Client instance in the process via the prototype chain, including ad-hoc constructions and any future Client paths.

Key implementation details:

- `discoverBunLockedDeps(prefix)` reads `node_modules/.bun/` filtered by prefix. Decodes Bun's `@langchain+core@1.1.40` -> `@langchain/core` package-name encoding (uses `lastIndexOf("@")` to handle scoped names; the naive `split("@")[0]` returns empty string for scoped packages -- found and fixed mid-implementation).
- `patchAllLangSmithClientPrototypes()` walks every `langsmith@*` copy in the bun graph, imports `dist/client.{cjs,js}`, and overrides `Client.prototype.processInputs` and `Client.prototype.processOutputs` to call `trimLargeValues`.
- `patchClientPrototype` is idempotent via a module-scope `WeakSet<object>` keyed on the prototype object.
- `MAX_FIELD_BYTES` lowered from 20MB -> 18MB to leave headroom for multipart-encoding overhead under the 25 MiB server cap.
- `MAX_VALUE_BYTES` (new, 24MB) enforces a per-key byte cap inside `trimLargeValues`. Even if a record has a single key whose value exceeds the cap, that key gets replaced with `[truncated: N bytes]` regardless of record total. Defends against the captured payload's exact shape (`{output: <75MB>}`).
- `trimLargeValues` widened to accept `unknown` and pass through `null`/`undefined`/non-record inputs unchanged. The langsmith SDK at `client.js:1793-1796` calls processInputs/processOutputs on raw `run.inputs`/`run.outputs` which can be null or non-record types; v3 hit a TypeError at `JSON.stringify(undefined).length` before this guard was added.
- `safeJsonSize` helper handles the `JSON.stringify(undefined) === undefined` edge case.
- Internal helpers exposed via `_internal` for unit tests only.

### `packages/agent/src/langsmith.test.ts` (new)

9 unit tests against `_internal`:

- Returns input unchanged when total under `MAX_FIELD_BYTES`.
- Returns null/undefined/string/array inputs unchanged (the SDK passes raw `run.inputs`/`run.outputs` which can be any of these).
- Handles records with `undefined` values without throwing on `JSON.stringify(undefined).length`.
- **Single-key value over `MAX_VALUE_BYTES` gets truncated even when record total is below threshold.** This is the SIO-687-shaped test.
- Truncates the largest field first when total exceeds `MAX_FIELD_BYTES`.
- Falls back to whole-record truncation when no per-key trim brings total under budget.
- Preserves a small field even after another is truncated.
- Handles the captured SIO-687 payload shape `{output: <single huge ToolMessage>}` directly.
- Asserts `MAX_FIELD_BYTES < SERVER_FIELD_LIMIT` with sufficient multipart-overhead headroom.

## Why this design

- **Patch the prototype, not instances.** Per-instance patching only covers Clients we know about at boot time. Tool-execution paths in LangGraph construct new RunTrees with reassigned `client` fields and bypass singleton-only patches. Patching `Client.prototype.processInputs/processOutputs` ensures every Client instance ever constructed picks up the override via prototype lookup.
- **Walk the filesystem to find every `langsmith@*` copy.** The bun graph has multiple langsmith copies because of peer-dep version skew (top-level `langsmith@0.6.3`, nested `langsmith@0.5.12` under older `@langchain/core`). Each copy defines its own `Client` class with a separate prototype. Hardcoding `require.resolve("langsmith")` only patches one.
- **Patch CJS *and* ESM at every version.** Bun's dual-package hazard means CJS and ESM imports of the same file produce separate module objects with separate `Client` classes -- each prototype must be patched independently.
- **`WeakSet<object>` keyed on prototype for idempotency.** Multiple discovery rounds may find the same prototype object via different paths. WeakSet lets us no-op without growing a Set indefinitely.
- **Per-key cap (`MAX_VALUE_BYTES`) defends in depth.** Even if some future Client constructor pulls in a langsmith copy outside our discovery prefix (e.g. a vendored copy), the per-key cap inside `trimLargeValues` ensures any single value above the cap is truncated -- IF the patched prototype is reached. The cap is inside `trimLargeValues` itself, which runs only when `processInputs/processOutputs` does, so this defense activates only on patched paths.
- **Lower `MAX_FIELD_BYTES` 20MB -> 18MB.** The 422 on a 75MB field is unambiguous; the 26MB-cap-with-21MB-JSON edge case is more subtle. 18MB JSON gives ~7MiB of multipart-encoding headroom.
- **Built-in SDK capture instead of custom instrumentation.** `langsmith@0.6.3/dist/client.js:1782-1786` already writes failed multipart bodies to `failedTracesDir` via `Client._writeTraceToFallbackDir` -- triggered by `LANGSMITH_FAILED_TRACES_DIR` env var. Reusing existing infrastructure for diagnosis was much smaller than the original plan's "wrap _sendMultipartRequest with logging" approach.
- **Trim function widened to `unknown` arg.** The langsmith SDK at `client.js:1793-1796` calls `processInputs`/`processOutputs` on raw `run.inputs`/`run.outputs` which can be null/undefined/non-record. v3 hit a runtime TypeError at `JSON.stringify(undefined).length` before this guard was added; failure mode was per-trace `Error in patchRun for run <id>` log lines, not 422s, but it broke trace upload entirely on those runs.

## Gotchas (carry forward)

Mostly inherited. New gotchas from this session marked **NEW**.

1. **NEW: 422 capture via SDK env vars.** `LANGSMITH_FAILED_TRACES_DIR=/tmp/dump` + `LANGSMITH_FAILED_TRACES_MAX_MB=500` in `.env` triggers the langsmith SDK's built-in failed-trace capture (`client.js:1782-1786`). On 422, the rejected multipart body is base64-encoded and written to `<dir>/trace_<ts>_<uuid>.json`. Decode with the multipart parser in `/tmp/sio687-decode-capture.py` (kept session-local; recreate from this session's git diff if needed).

2. **NEW: Bun's `node_modules/.bun/<pkg>+<ns>@<ver>` layout decoding.** To go from a directory name like `@langchain+core@1.1.40` back to `@langchain/core`, use `lastIndexOf("@")` to split off the version suffix, then replace `+` with `/` in the prefix. The naive `split("@")[0]` returns empty string for scoped packages and silently breaks the discovery walk.

3. **NEW: Multiple `@langchain/core` peer-dep variants are normal in this repo.** As of 2026-05-10, both 1.1.40 (workspace dep) and 1.1.39 (resolved peer of `@langchain/aws`) coexist. Don't try to dedupe via package.json overrides -- treat the multi-copy reality as the contract.

4. **NEW: Patch Client.prototype, not Client instances.** v1/v2 verification with the instance-walker showed `LangChainTracer.getRunTreeWithTracingConfig` constructs new RunTrees with reassigned `client` fields, bypassing singleton patches. The fix landed in this PR patches `Client.prototype.processInputs` and `Client.prototype.processOutputs` directly so every Client constructed via `new Client(...)` -- past, present, future, ad-hoc -- shares the override via prototype lookup.

4. **Run eval scripts from workspace root.** Bun's `.env` auto-discovery doesn't walk up from a sub-package cwd. (Inherited.)

5. **Localhost precheck blocks AgentCore-hosted MCPs (konnect).** Recreate `packages/agent/src/eval/run-eval-no-precheck.ts` from `run-eval.ts` template; **do NOT commit it** (now also enforced via `.gitignore` in this session). (Inherited; gitignore is new.)

6. **Bun process hangs post-`Done.` on LangSmith trace upload.** Safe to `kill <pid>` after `Done.` prints. (Inherited.)

7. **`MCP_OAUTH_HEADLESS=true` in `.env`** from SIO-693; no popup loops this session. (Inherited.)

8. **`bun run dev` fan-out to all workspace packages.** Couchbase :9082, gitlab :9084, atlassian :9085, elastic :9080; kafka via AgentCore proxy; konnect doesn't bind locally. (Inherited.)

9. **Trace categorization for sub-agent runs requires the langsmith Python SDK,** not just `langsmith-fetch` CLI. CLI's `traces` subcommand returns trace summaries (no outputs); SDK's `Client.list_runs(project_name=, is_root=...)` is needed for run-level outputs and the deep walk used in this session's investigation. Use the venv at `/Users/Simon.Owusu@Tommy.com/.local/pipx/venvs/langsmith-fetch/bin/python`. (Inherited; pattern reused this session.)

10. **422'd payloads are NOT persisted server-side.** When LangSmith returns 422, the run record's `outputs` field never gets the oversized data. Querying the project for "runs with outputs > N" will not find the offending runs -- they exist only as a server-rejected upload. **The only way to inspect a 422'd payload is the SDK-side `LANGSMITH_FAILED_TRACES_DIR` capture.** (NEW; learned the hard way during phase 1 of this session when the CLI returned 0 oversized runs across 965 scanned.)
