## What's broken

During SIO-688 pass-4 eval, the elastic sub-agent hit the LangGraph default recursion limit of 25 on the query that produced 6 `elasticsearch_search` truncations. Sub-agent ran 202s before failing.

```
4:50:58 PM error: Sub-agent failed
  service: agent:sub-agent
  dataSourceId: elastic
  isRetry: false
  duration: 202609
  error: "Recursion limit of 25 reached without hitting a stop condition.
          You can increase the limit by setting the \"recursionLimit\" config key."
  requestId: 095a4800-f737-4b74-b4d9-68780f96ba5e
```

This is **not** a SIO-688 truncator regression — every truncation in pass-4 correctly used a json-* strategy (100% mix, 9/9). The model is iterating heavily on multi-turn search refinement, and the better json-array preservation likely gives it enough partial signal to keep iterating rather than settle.

|---|---|---|---|

### Step 1: Pull the failing trace from LangSmith

```bash
grep "^LANGSMITH_API_KEY=" .env
LANGSMITH_API_KEY=<key> LANGSMITH_PROJECT=devops-incident-analyzer langsmith-fetch traces /tmp/sio689-traces \
  --limit 50 --include-metadata --format pretty
```

Filter for experiment `agent-eval-sio688-pass4-ba31aaf-c3ab89a1`, request `095a4800-f737-4b74-b4d9-68780f96ba5e`. The 25 tool calls on the failing elastic sub-agent are what need inspection.

### Step 2: Categorize the 25 tool calls

Three categories to count:

- **Distinct queries** (different index, different filter, different time window): the model is doing legitimate refinement and a higher limit would help.
- **Near-duplicates** (same query with minor parameter tweaks, same result coming back truncated): the model is looping; raising the limit just delays the failure.
- **Progressive refinement** (same query domain but narrowing toward an answer): higher limit is correct, but a budget hint in the prompt would also help.

The mix determines which of the three remediations below is right. Don't pick one before doing this categorization.

### Step 3: Pick one remediation

Three options, listed roughly in increasing implementation cost:

1. **Raise the limit for elastic specifically.** `packages/agent/src/sub-agent.ts` is the ReAct invocation site — check whether `recursionLimit` is configurable in the LangGraph config object and pass `40` (or whatever step 2 evidence supports) when `dataSourceId === "elastic"`. Quick to ship; may just delay the symptom if step 2 shows looping.
2. **Iteration-budget prompt hint.** Inject "you have N tool calls remaining" into the sub-agent system prompt or a synthetic observation between iterations. Lets the model self-pace. Medium effort; need to find an injection point in the ReAct loop.
3. **Near-duplicate detection short-circuit.** Hash recent tool-call args, when a near-duplicate fires, replace the result with a synthetic "you already tried this query — try a different approach" observation. Highest effort; worth it only if step 2 shows the failure mode is genuine looping, not refinement.

### Step 4: Verify

- Re-run the failing query (or a comparable multi-search incident) and confirm no recursion error.
- Run the standard 5-query eval to confirm no regression in the other 4 queries' confidence or truncation mix.
- `bun run typecheck && bun run lint && bun run --filter '@devops-agent/agent' test` all green.

## Gotchas inherited from prior sessions

These all carry forward and are still load-bearing:

1. **Run eval scripts from workspace root.** `bun --env-file .env packages/agent/src/eval/run-eval.ts` — Bun's `.env` auto-discovery doesn't walk up from a sub-package cwd.
2. **`MCP_OAUTH_HEADLESS=true` lives in the MCP server process.** If gitlab/atlassian popups appear during a "headless" eval, restart the MCP server with the flag in *its* env.
3. **`tail -N` buffers stdout.** Pipe straight to a file (`> /tmp/run.log 2>&1`), never through `tail`.
4. **Localhost precheck blocks AgentCore-hosted MCPs.** If kafka isn't running locally, recreate `run-eval-no-precheck.ts` from the snippet in `HANDOFF-2026-05-09-sio-686-eval-sizing.md` (do NOT commit it).
5. **Bun process hangs post-`Done.` on LangSmith trace upload.** Safe to `kill <pid>` after `Done.` prints — experiment data is already submitted.
6. **Confidence scores are LLM-judge stochastic.** Single-run deltas are noise; multi-pass averages are signal.
